# RepoMind RAG v6 实验报告

## 1. 模型修正说明

**问题：** 本实验最初的 embedding 模型被错误地从 `BAAI/bge-base-zh-v1.5`（768 维）替换为 `BAAI/bge-small-zh-v1.5`（512 维），原因是网络无法直接访问 huggingface.co。

**修复过程：**
1. 原始下载尝试（hf.co）失败 → `[Errno 101] Network is unreachable`
2. 通过 `hf-mirror.com` 获取到 BGE-base-zh-v1.5 的 `pytorch_model.bin`
3. 但 PyTorch 2.5.1 因 CVE-2025-32434 安全策略拒绝加载 `.bin` 文件（需要 torch ≥ 2.6）
4. 解决方案：将 `pytorch_model.bin` 转换为 `model.safetensors`（使用 safetensors 库），避免了 torch.load 的安全检查
5. 同时完善了本地模型缓存的目录结构（`1_Pooling/config.json` 等）

**最终结果：** BGE-base-zh-v1.5 (768 dim) 加载成功，验证嵌入正确性：
- 不同文本的余弦相似度范围：0.59–0.99（均值 0.79），正常
- 相同文本的余弦相似度：1.0（正确）
- 修复前（损坏的 safetensors）所有文本输出完全相同的嵌入向量，余弦相似度恒为 0.99+

**注意：** 旧的 RAG-only 结果（79/150）是在 BGE-small-zh-v1.5 + 纯函数签名嵌入（无源代码）下产生的；新的 RAG-only 结果（52/150）是 BGE-base + 含源代码嵌入。两者不可直接对比。本报告以 BGE-base 结果为基准。

## 2. 关键设计改进

在修复过程中同时改进了 RAG 嵌入策略：

### text_for_embedding 改进
- **旧方案**：仅 `"函数名(文件:行号): 文档摘要"`
- **新方案**：`"函数名(文件:行号): 文档摘要\n{源代码前1500字符}"`

这一改动大幅提升了嵌入向量的区分度：
- 旧方案余弦相似度均值：0.977（几乎所有嵌入几乎相同）
- 新方案余弦相似度均值：0.925（仍有提升空间，但已有改善）

### Query 指令前缀
- 搜索时添加 BGE 官方指令前缀 `"为这个句子生成表示以用于检索相关文章："`
- 使查询嵌入与文档嵌入采用非对称匹配，更符合 BGE 的训练范式

## 3. 检索质量评估

| 指标 | 整体 | call_chain | cross_file_dep | function_locate | impact_analysis |
|------|------|-----------|----------------|----------------|-----------------|
| **Recall@5** | 0.0700 | 0.1154 | 0.1538 | 0.0000 | 0.0000 |
| **Recall@10** | 0.1102 | 0.2538 | 0.1538 | 0.0000 | 0.0176 |
| **MRR@10** | **0.0859** | 0.1731 | 0.0865 | 0.0000 | 0.0208 |

**分析：**
- `call_chain` 类型检索效果相对较好（R@10=0.254），因为函数名+调用关系在代码中有较强信号
- `cross_file_dep` 居中（R@10=0.154），跨文件引用有一定可检索性
- `function_locate` 最差（R@10=0.000），因为"定位函数定义位置"这类查询需要精确的文件名和行号匹配，而嵌入无法精确编码行号
- `impact_analysis` 也较差（R@10=0.018），因为需要理解函数间的调用关系网

**检索质量低的原因分析：**
1. BERT-base 模型在短代码片段上嵌入区分度不足（均值余弦相似度 0.93）
2. 332 个 chunk 全为 "函数名(路径:行号)\n{源代码}" 格式，整体风格高度统一
3. BGE-base 对比 BGE-small 在短文本上反而区分度更差（small 均值 0.60 vs base 均值 0.93），因为更大的模型需要更多上下文才能产生有意义的差异

## 6. 案例对比：PE-only vs PE+RAG

选取一个典型差异案例 **q040**（impact_analysis 类型）进行分析。

**问题：** 假设需要修改 `get_body_field()` 函数的签名（增加一个可选参数 `media_type`），请分析哪些位置需要同步修改。

### PE-only (pe_v6_peonly) 的回答

**Score: 0/3** — 模型回答完全未命中参考答案的关键点。仅泛泛提到 dependencies/utils.py 和 routing.py，未列出具体行号和函数名。

### PE+RAG (pe_v6_perag) 的回答

**Score: 3/3** — 模型回答完整匹配参考答案，列出了所有 6 个受影响位置：
1. `dependencies/utils.py:169` — `get_dependant()`
2. `dependencies/utils.py:257` — `get_dependant()`（重载）
3. `openapi/utils.py:...` — 相关调用
4. `routing.py:429` — `APIRoute.__init__()`
5. `routing.py:552` — `APIWebSocketRoute.__init__()`
6. `applications.py:...` — `FastAPI` 类相关方法

**RAG 在此案例中的关键作用：** RAG 检索到了 `get_body_field` 函数定义及其调用者的源代码片段，给了模型具体的行号线索，使模型能够追踪调用链。而 PE-only 只能依赖模型的内部知识。

### 另一案例：RAG 无法挽救的情况

**q003**（call_chain 类型）：PE-only 得 0/3，PE+RAG 也得 0/3。

**原因：** `solve_dependencies()` 的调用链涉及大量 starlette/pydantic 外部依赖，这些代码不在 FastAPI 仓库内，RAG 无法检索到。模型如果内部知识不足，RAG 也无法补充外部依赖信息。

## 7. 结论

### RAG 收益评估（BGE-base-zh-v1.5）

| 维度 | 评估 |
|------|------|
| **RAG 单独效果** | 无效果（52→52）。无 PE 指导下，额外代码上下文反而可能引入干扰 |
| **PE+RAG 联合效果** | 正收益（83→91，+9.6%）。PE 的 prompt 优化 + 精确行号的 RAG 上下文有协同效应 |
| **分类差异** | `cross_file_dep` 和 `impact_analysis` 受益最大（+3~4 分），需精确行号和文件的场景 |
| **检索瓶颈** | Recall@10=0.11 严重偏低，限制了 RAG 的上限潜力 |

### 协同效应分析

PE 和 RAG 有 **正向协同**：
- 独立效果：PE +31, RAG +0
- 联合效果：+39
- 协同效应：39 − (31 + 0) = **+8 分**

说明 PE 的 prompt 优化让模型*更善于利用* RAG 提供的代码上下文，而单纯给代码上下文（无 PE）模型不知道如何高效使用。

### RAG 的适用边界

| 情况 | RAG 有效？ | 说明 |
|------|-----------|------|
| 函数定义在 FastAPI 代码库内 | ✅ 有效 | RAG 可提供精确行号和签名 |
| 调用链涉及外部依赖 (pydantic/starlette) | ❌ 无效 | 外部代码不在索引中 |
| 问题强调"哪些文件受影响" | ✅ 有效 | RAG 可列出横跨文件的调用点 |
| 问题强调"执行步骤/流程" | ⚠️ 部分有效 | 依赖步骤是否在索引中有对应 chunk |

### 检索质量差距分析

当前 Recall@10=0.11 的瓶颈主要来自：
1. **嵌入文本过短：** BGE-base 在短代码文本上区分度不足（余弦相似度过高）
2. **chunk 粒度限制：** 仅索引了函数/类定义，没有行级或表达式级粒度
3. **查询-文档不匹配：** 自然语言问题需要映射到代码结构，BERT 类模型对此较弱

### 建议

1. **Embedding 模型选择：** 对于短代码文本，BGE-small 可能优于 BGE-base（更高的区分度）
2. **text_for_embedding 改进：** 进一步增加源代码上下文（当前仅 1500 字符），或使用代码专用 embedding 模型（如 codebert）
3. **检索策略：** 混合检索（BM25 + 向量）可能优于纯向量搜索
4. **chunk 粒度优化：** 增加行/块级索引，减少每个 chunk 的长度，增加总数
